# Reasoning-First Protocol — Design Document

> **Status:** v1.1 implemented (opt-in). Default OFF; preserves legacy behavior.
> **Author:** Super Z (autonomous development pass, 2026-06-30)
> **Goal:** Turn Hermes-Omni into a reasoning-first agent where the framework
> *enables* the LLM's intelligence instead of bottlenecking it, while
> preventing reckless actions through context-aware safety (not regex).
> **v1.1 additions:** Long-run engine (background reflection + checkpoint/resume),
> Tool router (auto tool selection).

---

## 1. Problem Statement

The fork `airdropp20208-star/hermes-omni` has two structural problems that
this design addresses:

### 1.1 Unlimited mode is the wrong kind of safety removal

The "unlimited mode" commits (`be3646d`, `0d1038f`) disabled *all* safety
guardrails — pattern-based blocking, approval prompts, rate limiting, file
size caps, context window floors, idle timeouts. The stated goal was "agent
runs like Codex" but the implementation is "agent has no safety at all":

- `rm -rf /` executes without question
- `MINIMUM_CONTEXT_LENGTH = 1` accepts models that will immediately OOM
- Tool loop detection is set to 999,999 — agent can call the same failing
  tool forever
- File read cap is 999,999,999 chars — single read can OOM the process

This is **not** what a reasoning-first agent needs. A reasoning-first agent
needs *smart* safety — safety that understands context and scales with the
LLM's intelligence. Removing all safety just makes the agent reckless.

### 1.2 Pattern-based guardian cannot reason

The legacy guardian (`agent/unified/policy.py`, `agent/guardian.py`) blocks
actions using regex patterns. This is the wrong abstraction:

- `rm\s+-rf` matches both `rm -rf node_modules/` (safe) and `rm -rf /` (catastrophic). Regex cannot tell them apart.
- The rules never improve. Once written, they're static.
- Creative workflows get over-blocked. A senior engineer doing
  `git push --force` to their own feature branch knows what they're doing.

### 1.3 Two parallel OmniAgent integration layers

The fork has two overlapping integration layers:

| Layer | Path | Wiring | Storage |
|---|---|---|---|
| A (declarative) | `agent/unified/` | `model_tools.handle_function_call` | `~/.hermes/unified/reflexions.jsonl` |
| B (port-imperative) | `agent/omni_integration.py` | `conversation_loop.run_conversation` | `.hermes/learnings/lessons.jsonl` |

Both implement guardian + reflexion with different APIs. This is tech debt.

## 2. Design Goals

1. **Don't bottleneck the LLM.** Plans are generated by the LLM itself,
   through a single function call. No multi-turn scaffolding, no chained
   prompts. One call, one plan.

2. **Match plan depth to action consequence.** A `read_file` is TRIVIAL —
   skip planning entirely. A `git push --force` is IRREVERSIBLE — generate
   a full plan AND a critique AND require explicit acknowledgement.

3. **Never block — escalate.** The protocol does not refuse actions. It
   returns structured reasoning that the conversation loop can choose to
   present to the user, log, or proceed. The LLM is the judge, not the
   framework.

4. **Fail open, not closed.** If the LLM call fails (network, quota, etc.),
   the protocol degrades to the legacy path: execute the tool. Better to
   execute one risky action than to brick the agent entirely.

5. **Opt-in, not default.** All new behavior is gated behind config flags.
   Default behavior is unchanged. Users opt into reasoning-first mode
   explicitly.

6. **Build on the cleaner integration layer.** All new code lives under
   `agent/unified/`. The legacy `agent/omni_integration.py` is marked
   deprecated but not removed (to avoid breaking `conversation_loop.py`).

## 4. Decision Classification

The `DecisionFramework` (`agent/unified/decision.py`) classifies every tool
call into one of four classes. Classification is pure-Python heuristics
(regex + tool-name matching), runs in microseconds, and never blocks.

| Class | Examples | Behavior |
|---|---|---|
| **TRIVIAL** | `read_file`, `ls`, `grep`, `glob`, `unified_recall` | Execute immediately, no plan |
| **STANDARD** | `bash ls`, unknown tools, default fallback | Brief plan (goal + approach), execute |
| **CONSEQUENTIAL** | `git push`, `sudo`, `pip install`, `INSERT INTO`, `curl \| sh` | Full plan + critique, execute if sound |
| **IRREVERSIBLE** | `rm -rf /`, `mkfs`, `dd if=...of=/dev/`, `git push --force`, `DROP TABLE`, `shutdown` | Full plan + critique + explicit acknowledgement (hard floor by default) |

Classification rules:

1. **Argument values override tool names.** `bash` with `command="ls"` is
   STANDARD; `bash` with `command="rm -rf /"` is IRREVERSIBLE. The risk is
   in the value, not the tool.
2. **Read-only tools default to TRIVIAL.** No plan needed for inspection.
3. **Send/publish/deploy tools default to CONSEQUENTIAL.** External side
   effects are non-trivial.
4. **Execute/bash/run tools default to STANDARD.** Unless args escalate.
5. **Unknown tools default to STANDARD.** Conservative — better to plan
   once too many than once too few.

## 5. Reasoning Protocol

The `ReasoningProtocol` (`agent/unified/reasoning.py`) is a 4-phase loop:
**plan → critique → execute → reflect**.

### 5.1 Plan

Generated by a single LLM call. Returns STRICT JSON:

```json
{
  "goal": "what this action is supposed to achieve",
  "approach": "how this tool call achieves the goal",
  "alternatives": "other viable approaches, or empty string if none",
  "risks": "what could go wrong, or empty string if nothing notable",
  "reversibility": "can this be undone? how?",
  "decision": "proceed" | "abort" | "ask_user",
  "rationale": "one-sentence justification for the decision"
}
```

If `decision` is `abort`, the protocol blocks the action with the rationale.
If `decision` is `ask_user`, the protocol surfaces the rationale to the LLM
(which can choose to surface it to the user).

### 5.2 Critique

Only generated for CONSEQUENTIAL+ actions. Same LLM, different prompt:

```json
{
  "blind_spots": "what the plan missed",
  "assumptions_verified": "assumptions the plan made that are correct",
  "assumptions_invalid": "assumptions the plan made that are wrong, or empty",
  "recommendation": "proceed" | "modify" | "abort",
  "modified_approach": "if recommendation=modify, describe the change"
}
```

The critique is currently advisory — it's emitted as an event and the LLM
sees it in the conversation. A future version may auto-block on
`recommendation=abort`.

### 5.3 Execute

The actual tool call. This is handled by Hermes' existing tool dispatcher;
the protocol doesn't touch it.

### 5.4 Reflect

After execution, the protocol generates a reflection:

```json
{
  "outcome_matched": true | false,
  "expected": "what the plan expected",
  "actual": "what actually happened (brief)",
  "lesson": "the transferable lesson, or empty if nothing notable",
  "score": 0.0 to 5.0
}
```

If `lesson` is non-empty, it's persisted into the `ReflexionStore` as a
record with `source="reasoning_reflection"` and `tags=["reflection",
"reasoning"]`. Future similar situations can recall it.

### 5.5 Fail-open semantics

Every LLM call is wrapped in `try/except`. If the LLM is unavailable,
returns garbage, or raises:

- `plan()` returns `None` → action proceeds without plan
- `critique()` returns `None` → action proceeds without critique
- `reflect()` returns `None` → no reflection persisted
- `SmartGuardian.assess()` returns `WARN` → action proceeds with warning

The agent never gets bricked by an LLM outage.

## 6. Smart Guardian

The `SmartGuardian` (`agent/unified/smart_guardian.py`) is the
LLM-as-judge replacement for the pattern-based guardian.

### 6.1 Pipeline

1. Pattern-based `PolicyEngine` runs FIRST (legacy). Blocks catastrophic
   patterns (`rm -rf /`, fork bomb, `mkfs`) with zero LLM cost.
2. `DecisionFramework` classifies the action.
3. If TRIVIAL or STANDARD → return `ALLOW` immediately (skip judge).
4. If CONSEQUENTIAL or IRREVERSIBLE → call LLM judge (with caching).
5. If LLM unavailable → return `WARN` (fail open).

### 6.2 Verdicts

| Verdict | Behavior |
|---|---|
| `ALLOW` | Proceed, no warning |
| `WARN` | Proceed, but log/notify |
| `REQUIRE_USER_CONFIRM` | Pause and ask the user |
| `BLOCK` | Refuse; this is final |

### 6.3 Caching

Verdicts are cached by a content-hash of (tool_name, args,
classification_reason, plan_decision). Cache has:

- **Bounded size** (default 512 entries, LRU eviction)
- **TTL** (default 1 hour, so stale verdicts don't permanently allow
  risky patterns)

The cache key intentionally excludes the plan *text* (which varies with
LLM temperature) — only the plan's *decision* field is included. Two
plans with the same decision but different prose share a cache entry.

### 6.4 Hard floor for IRREVERSIBLE

By default (`hard_block_irreversible: true`), IRREVERSIBLE actions are
blocked at the framework layer even if the LLM judge says `ALLOW`. The
block message tells the LLM to either:

1. Call the `reasoning_decide` tool with `acknowledged_irreversible=true`
   and a rationale, OR
2. Choose a safer alternative.

This is the **bridge between "unlimited mode" (no safety) and "pattern-block
mode" (regex can't reason)**. The agent has the *appearance* of unlimited
autonomy (it can do anything with explicit acknowledgement) while
maintaining a context-aware safety net that scales with the LLM's
intelligence.

Set `unified.smart_guardian.hard_block_irreversible: false` to disable
the hard floor entirely (true Codex-style autonomy — trust the LLM judge
completely).

## 7. Explicit Reasoning Tools

Four new tools in `tools/reasoning_tools.py` let the agent *explicitly*
express its reasoning to the framework, even when the implicit protocol
is disabled:

| Tool | Purpose |
|---|---|
| `reasoning_plan` | Record an explicit plan before a consequential action |
| `reasoning_critique` | Record a self-critique of a plan |
| `reasoning_decide` | Make an explicit, auditable decision (with optional `acknowledged_irreversible=true`) |
| `reasoning_reflect` | Record a post-action reflection (lesson persisted to reflexion store) |

These tools exist because:

1. **Transparency.** Users can see *why* the agent chose to act.
2. **Override hook.** The agent can call `reasoning_decide` with
   `decision=ask_user` when genuinely unsure.
3. **Persistent record.** Plans/decisions are logged in conversation
   history and can be replayed/audited.
4. **Works with any LLM.** Models that don't natively emit structured
   reasoning can be prompted to call these tools explicitly.

The `reasoning_decide` tool is the **escape hatch** for the hard floor.
When the framework blocks an IRREVERSIBLE action, the LLM can call
`reasoning_decide` with `acknowledged_irreversible=true` to proceed. This
makes the agent's reasoning auditable — every irreversible action has a
recorded rationale.

## 8. Configuration

All new behavior is opt-in via `config.yaml`:

```yaml
unified:
  enabled: true                    # master switch (existing)
  reflexion:
    enabled: true                  # existing
  guardian:
    enabled: true                  # existing (pattern-based)
    block_tools: []                # existing

  # --- New: Reasoning-first protocol v1 ---
  reasoning:
    enabled: false                 # default OFF
    persist_reflections: true      # persist lessons into reflexion store

  smart_guardian:
    enabled: false                 # default OFF
    hard_block_irreversible: true  # hard floor for IRREVERSIBLE
    cache_size: 512
    cache_ttl_seconds: 3600
```

Environment variable overrides:

```bash
export HERMES_UNIFIED_REASONING=1
export HERMES_UNIFIED_SMART_GUARDIAN=1
```

### 8.1 Wiring the LLM

The reasoning protocol and smart guardian need an LLM client to generate
plans/verdicts. Hermes' setup path must call this once the LLM client is
available:

```python
from agent.unified.integration import configure_reasoning_stack

def llm_call(system_prompt: str, user_prompt: str) -> str:
    return my_llm_client.chat(system=system_prompt, user=user_prompt)

configure_reasoning_stack(
    llm_call=llm_call,
    conversation_context_provider=lambda: get_recent_context_summary(),
)
```

Until this is called, the protocol degrades to no-LLM mode (skips plans,
returns WARN for consequential actions).

### 8.2 Recommended presets

**Conservative (default):**
```yaml
unified:
  reasoning:
    enabled: false
  smart_guardian:
    enabled: false
```
Behavior identical to legacy Hermes. No LLM cost overhead.

**Reasoning-first (recommended for production):**
```yaml
unified:
  reasoning:
    enabled: true
    persist_reflections: true
  smart_guardian:
    enabled: true
    hard_block_irreversible: true
```
Agent reasons before every consequential action, LLM judge evaluates risk,
hard floor prevents irreversible actions without explicit acknowledgement.

**Full autonomy (Codex-style, NOT recommended):**
```yaml
unified:
  reasoning:
    enabled: true
  smart_guardian:
    enabled: true
    hard_block_irreversible: false  # trust the LLM judge entirely
```
Agent reasons and the LLM judge is the only safety net. Closest to
"unlimited mode" but with structured reasoning — the LLM must justify
every consequential action, and the justification is auditable.

## 11. Limitations & Future Work

### 11.1 Not yet wired into the conversation loop

The reasoning protocol is wired into `model_tools.handle_function_call()`
via `agent.unified.integration.before_tool_call()`/`after_tool_call()`.
But the LLM client is not yet wired in — `configure_reasoning_stack()`
needs to be called from Hermes' setup path. This is a one-line change in
`hermes_cli/setup.py` or `agent/conversation_loop.py` but requires
knowing where the LLM client is initialized.

### 11.2 Critique is advisory, not enforcing

The critique phase currently emits an event but does not auto-block on
`recommendation=abort`. This is intentional for v1 (don't over-block
during rollout) but should be tightened in v2.

### 11.3 No streaming for plan generation

The plan LLM call is synchronous. For top-tier models with slow planning
(e.g., Claude Opus with extended thinking), this could add 5-15 seconds
to consequential actions. Streaming would improve UX.

### 11.4 No multi-model routing

The protocol uses whatever LLM is configured. A future version could
route plan generation to a fast model (Haiku/MiniMax) and risk
assessment to a strong model (Opus/GPT-5), optimizing cost.

### 11.5 No reflexion on success

Currently, reflections are only persisted if `lesson` is non-empty. The
LLM is instructed to return empty lesson for unremarkable successes.
This means we don't learn from successes — only from mismatches. A
future version could persist "what worked" lessons too.

### 11.6 Conversation context provider not implemented

The `conversation_context_provider` parameter exists but is not
implemented in this PR. The protocol falls back to empty context. A
future version should summarize the last N turns and feed it to the
planning prompt.

### 11.7 Legacy OmniFeatureGate not migrated

`agent/omni_integration.py` is marked deprecated but still imported by
`conversation_loop.py`. Migration requires touching the mega-file
`conversation_loop.py` (4,843 lines) — risky. Defer to a separate PR.

## 12. Why This Approach (vs. Alternatives)

### vs. "Just use unlimited mode"

Unlimited mode removes all safety. This works for trusted environments
but is reckless for production. The reasoning-first protocol provides
*smarter* safety — safety that scales with the LLM's intelligence.

### vs. "Just use pattern-based guardian"

Pattern-based guardian cannot reason. It blocks `rm -rf` regardless of
target, which over-blocks legitimate work. The Smart Guardian uses the
LLM to evaluate context, so `rm -rf node_modules/` is allowed while
`rm -rf /` is blocked.

### vs. "Just rely on the LLM's internal chain-of-thought"

Top-tier LLMs reason well internally, but:

- Not all models do. The framework should support any model.
- Internal CoT is invisible — can't be persisted, recalled, or audited.
- The protocol gives a *structured* reasoning artifact for downstream use.

The protocol is NOT a replacement for the LLM's own reasoning — it's a
structured surface that *captures* that reasoning.

### vs. "Multi-agent debate"

Multi-agent debate (e.g., 3 agents arguing) is expensive and slow. The
reasoning protocol uses a single LLM call per phase, which is 3-10x
cheaper than debate. For most consequential actions, single-call
reasoning is sufficient.

### vs. " Constitutional AI / RLHF"

Constitutional AI and RLHF are training-time interventions. The
reasoning-first protocol is inference-time — it works with any model
without retraining. It's complementary, not competing.

For users currently on unlimited mode who want to switch to
reasoning-first:

1. **Enable reasoning** (zero risk — protocol degrades to no-op if LLM

   ```yaml

     reasoning:
       enabled: true
   ```

2. **Wire the LLM** (one-line change in setup path):

3. **Enable Smart Guardian** (adds LLM cost for consequential actions

   ```yaml

     smart_guardian:
       enabled: true
   ```

   the reflexion store to see what's being learned.

5. **Optionally disable hard floor** once you trust the LLM judge:

   ```yaml
   unified:
     smart_guardian:
       hard_block_irreversible: false
   ```

# Part II — v1.1: Long-Run Engine & Tool Router

## 15. The Long-Run Problem

v1 added 1-3 LLM calls per consequential action (plan + critique +
reflection). This made the agent **smarter but slower**. For long-running
autonomous work — "monitor this repo and refactor when needed" — the
synchronous reflection after every tool call adds up:

- 50 tool calls × 2s reflection each = 100s of pure reflection overhead
- A single turn running for hours has no checkpoint — crash = total loss
- No way to enqueue "do X, then Y, then Z" across multiple ticks
- Reflection blocks the next tool call, increasing wall-clock time

The LongRunEngine solves all of these.

## 16. LongRunEngine Architecture

```
   ┌─────────────────────────────────────────────────────────┐
   │                     LongRunEngine                       │
   │                                                          │
   │  ┌──────────────┐    ┌──────────────┐    ┌───────────┐ │
   │  │  Work Queue  │ ─▶ │   Dispatcher │ ─▶ │  Worker   │ │
   │  │ (priority)   │    │  (priority)  │    │ (1 thread)│ │
   │  └──────────────┘    └──────────────┘    └───────────┘ │
   │         │                                       │       │
   │         ▼                                       ▼       │
   │  ┌──────────────┐                      ┌──────────────┐│
   │  │ Checkpoint   │ ◀── write after ────│  Result     ││
   │  │ Store (JSONL)│     each item done  │  Sink       ││
   │  └──────────────┘                      └──────────────┘│
   │                                                          │
   │  ┌──────────────────────────────────────────────────┐  │
   │  │  Background Reflection Worker (separate thread)  │  │
   │  │  - Debounced (waits 5s after last tool call)     │  │
   │  │  - Batched (groups up to 10 reflections)         │  │
   │  │  - Calls ReasoningProtocol.reflect_batch()       │  │
   │  └──────────────────────────────────────────────────┘  │
   └─────────────────────────────────────────────────────────┘
```

### 16.1 Key design decisions

1. **Single worker thread.** Avoids concurrency headaches with the tool
   registry (not thread-safe). One worker = no locking around tool dispatch.
2. **Priority queue.** Critical work (user-initiated) jumps ahead of
   background work (reflexion, indexing). Heap-based.
3. **Checkpoint after every item.** Each completed item is appended to a
   JSONL checkpoint file. On restart, the engine replays the checkpoint
   to skip already-done items (idempotency).
4. **Reflection is fire-and-forget.** The reflection worker runs on its
   own thread and never blocks tool dispatch. If the LLM call fails, the
   reflection is silently dropped (fail-open).
5. **Heartbeat.** The engine emits a heartbeat event every N seconds so
   monitoring can detect a stuck worker.
6. **No external deps.** Stdlib only (threading, queue, json, pathlib).
   No asyncio — would require rewriting the whole Hermes loop.

### 16.2 Background reflection worker

Debounced + batched:

- **Debounce:** waits `debounce_seconds` (default 5s) after the last
  reflection request before processing. Groups rapid-fire reflections
  (e.g., from a loop of tool calls) into a single batch.
- **Batch:** processes up to `batch_size` (default 10) items per LLM
  call. Uses `ReasoningProtocol.reflect_batch()` which sends all items
  in a single prompt and expects a JSON array back.
- **Fail-open:** if `reflect_batch_fn` raises or returns wrong shape,
  the whole batch is dropped (counted in `dropped` stat).

This turns 50 reflections × 2s each = 100s of synchronous overhead into
~5 batches × 3s each = 15s of background work that doesn't block the
agent at all.

### 16.3 Checkpoint/resume

The `CheckpointStore` is a JSONL append-only log at
`~/.hermes/unified/longrun_checkpoint.jsonl`. Each entry records:

```json
{
  "item_id": "test_task_42",
  "kind": "test_task",
  "started_at": 1738243200.0,
  "completed_at": 1738243200.5,
  "success": true,
  "result_preview": "processed test_task_42",
  "error": ""
}
```

On engine restart, the store is replayed to populate the
`_completed_ids` set. The dispatcher checks this set before processing
any item — duplicates are skipped silently. This means:

- Process crash mid-batch? Restart and the engine picks up where it
  left off.
- Network blip causes a duplicate enqueue? The second one is skipped.
- Want a fresh start? Call `engine.reset_checkpoint()`.

## 17. Tool Router — Auto Tool Selection

### 17.1 The problem

A senior engineer doesn't need to be told "use grep to search files" —
they just know. The same is true for LLMs:

- Top-tier models (Claude Opus, GPT-5, GLM-4.6) rarely need help.
- Mid-tier models often pick the wrong tool, miss available tools, or
  use generic `bash` when a purpose-built tool would work better.
- Even top-tier models miss tools they don't know exist — MCP servers,
  custom plugins, newly-registered skills.

### 17.2 Architecture

The ToolRouter:

1. **Indexes all available tools** from `tools.registry` — name,
   description, parameters, toolset, source.
2. **Routes by intent** — given a natural-language task description,
   returns the top-N tools most likely to be relevant. Uses BM25 first
   (cheap), then intent-pattern boosts (rule-based multipliers).
3. **Injects suggestions into the system prompt** — when enabled, the
   router prepends a `<tool-suggestions>` block so the LLM sees the
   suggestions before generating a tool call.
4. **Learns from usage** — counts how often each tool is actually
   called for a given intent. Frequently-used tools get a relevance
   boost in future routing.

### 17.3 Algorithm

For a given query:

1. Tokenize query (lowercase alphanumeric).
2. Compute BM25 score against every tool's tokenized text
   (name + description + parameter names).
3. Apply intent pattern boosts — 25 regex patterns detect common
   intents (read file, search, execute, web search, etc.) and multiply
   the BM25 score for matching tools.
4. Add usage bonus — for each query token, if the tool was previously
   called with that token in the query, add a small bonus (capped at
   +1.0 total).
5. Return top-N tools, formatted as a markdown block.

### 17.4 Intent patterns (excerpt)

| Pattern | Boosts |
|---|---|
| `read\|cat\|view file` | `read*` ×3.0, `file*` ×2.0, `cat*` ×2.0 |
| `search\|find\|grep in files` | `grep*` ×3.0, `search*` ×2.5, `find*` ×2.5 |
| `run\|execute\|bash\|shell` | `bash*` ×3.0, `shell*` ×2.5, `execute*` ×2.5 |
| `search\|google the web` | `web_search*` ×3.0, `search*` ×2.5 |
| `remember\|recall\|past experience` | `unified_recall*` ×3.0, `recall*` ×2.5 |
| `plan\|think\|reason before` | `reasoning_plan*` ×3.0 |
| `generate image\|picture` | `image_gen*` ×3.0 |
| `schedule\|cron\|every day` | `cron*` ×3.0 |
| `delegate\|subagent\|parallel` | `delegate*` ×3.0 |
| ... | (25 patterns total) |

```yaml
    enabled: false       # default OFF
```


### 17.7 Integration with existing tool_search.py

The existing `tools/tool_search.py` uses a "bridge tool" pattern to
defer tool loading — only the core tools are loaded initially, and a
`search_tools` bridge tool lets the LLM discover deferred tools on
demand. The ToolRouter is **complementary**:

- `tool_search.py` handles the **lazy loading** mechanics (don't load
  500 tool schemas until needed).
- `ToolRouter` handles **proactive suggestion** (tell the LLM which
  tools are relevant *before* it has to ask).

They can coexist: ToolRouter suggests the top-5 tools for the current
task, and `tool_search.py` still defers the rest. If the LLM follows
the suggestion, no bridge tool call is needed. If it doesn't, the
bridge tool is still available.

## 20. v1.1 Limitations & Future Work

### 20.1 Single worker thread

The LongRunEngine uses a single worker thread to avoid concurrency
issues with the tool registry. For I/O-bound work (network calls), this
limits throughput. A future version could use a thread pool for
I/O-bound handlers while keeping tool dispatch single-threaded.

### 20.2 No streaming for plan generation

Plan LLM calls are still synchronous in v1.1. For long plans (e.g.,
IRREVERSIBLE actions with full critique), this can block the agent for
5-15s. Streaming would improve UX.

### 20.3 Tool router has no LLM reranker

The router uses BM25 + rule-based intent patterns. An LLM reranker
(prompt: "given these 10 candidate tools and this task, which 3 are
most relevant?") would improve accuracy for ambiguous queries. Opt-in,
behind a config flag.

### 20.4 No automatic tool router refresh

The catalog is built once at first use. If tools are registered
dynamically (e.g., a new MCP server connects mid-session), the router
won't know about them until `refresh_router()` is called. A future
version could hook into `registry.register()` to auto-refresh.

### 20.5 Reflection worker doesn't retry

If `reflect_batch_fn` raises, the entire batch is dropped. A retry with
smaller batch size (e.g., split 10 into 2×5) would recover more
reflections. Trade-off: complexity vs. reflection value.

### 20.6 No persistence of usage histogram

The ToolRouter's usage histogram is in-memory only. On restart, the
learning is lost. A future version could persist it to
`~/.hermes/unified/tool_usage.json` alongside the reflexion store.
